# Обоснование выбора формата контейнера (этап 4)

## Базовый выбор: OCI/Docker-совместимый образ

Для проекта выбран **OCI-совместимый контейнерный образ** (де-факто стандарт, используемый Docker и большинством современных рантаймов).
Это упрощает:
- переносимость (Linux x86_64/arm64),
- включение в CI/CD пайплайны,
- развёртывание в Kubernetes/nomad и др. оркестраторах,
- использование альтернатив Docker Engine (при необходимости).

## Альтернативы Docker Engine (короткий обзор)

1. **Podman**
   - Запуск контейнеров без демона (daemonless), удобен в средах с требованиями безопасности.
   - Поддерживает OCI-образы и команды, близкие к Docker.

2. **containerd / nerdctl**
   - containerd — распространённый рантайм (часто используется как часть Kubernetes-стека).
   - nerdctl — CLI, похожий на docker, поверх containerd.

3. **Buildah / Kaniko (сборка образов)**
   - Buildah — сборка OCI-образов без демона.
   - Kaniko — сборка в Kubernetes/CI без привилегий (часто для безопасных пайплайнов).

4. **LXC/LXD (системные контейнеры)**
   - Ближе к «лёгким VM», полезно для системных сценариев.
   - Но для микросервисной доставки/CI чаще выбирают OCI.

## “Отечественные возможности” (как оформить в отчёте)

В отчёте можно кратко отметить, что в практических сценариях в РФ часто используются:
- ОС/платформы корпоративного класса на базе Linux (например, Astra Linux и др.),
- Docker-совместимые OCI-образы (чтобы сохранять переносимость),
- альтернативы Docker Engine (Podman / containerd) в зависимости от политики безопасности,
- отечественные репозитории/зеркала и закрытые registry для хранения образов.

**Вывод:** OCI-формат обеспечивает максимальную совместимость и минимальные риски при переносе проекта между средами и инструментами.
